# updated RAID log

_Produced by: RAID log maintenance_

**Business outcome supported:** produce working, tested, instrumented increments that meet the Definition of Done, with stakeholder-accepted quality and documented lessons for continuous improvement.

**Primary owner:** Project Manager

**Stakeholders:** _(none listed)_

**Accelerated MVP:** slim — the RAID log itself is already slim

## What this is

The Build-phase evolution of the RAID log set up in Plan — Risks, Assumptions, Issues, Dependencies maintained with fresh state as the initiative progresses. Evidence that governance is active, not just installed.

## Why it matters for the Project Manager

A stale RAID log is worse than none — it gives false confidence that things are tracked. Keeping the log honest through Build is how you preserve the early-warning system that separates a plan that ships from a plan that surprises.

## Definition of Done

- [ ] Items updated this cycle show state changes with evidence
- [ ] New items list owner, trigger, and mitigation
- [ ] Retired items record the outcome
- [ ] Aging indicators flag items stuck too long
- [ ] Summary metrics cover open, overdue, and aging counts

## What it contains

- Items updated this cycle with state changes and evidence
- New items added (with owner, trigger, mitigation)
- Retired items with outcome
- Aging indicators for items stuck too long
- Cross-references to change control package and risk register
- Summary metrics (open, overdue, aging)

## Inputs you rely on

- Original RAID log from Plan
- Cross-team sync notes and daily standups
- Architecture status, security findings, and incident signals
- Scope-change decisions
- Vendor and dependency updates

## Who consumes it

- Product Sponsor — reviews top items in steering and exec updates
- Architect and Security — own their technical/compliance items
- PMO and governance forums — cadence reviews
- Adjacent PMs — coordinate cross-initiative dependencies

## Common pitfalls

- Copy-paste updates with no real state change
- New risks not converted into mitigation tasks with owners
- Dependencies with "TBD" dates that never firm up
- Log ignored between status reports — updates are bulk and shallow
