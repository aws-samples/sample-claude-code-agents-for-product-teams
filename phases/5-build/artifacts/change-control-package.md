# change control package

_Produced by: Scope-change impact coordination_

**Business outcome supported:** produce working, tested, instrumented increments that meet the Definition of Done, with stakeholder-accepted quality and documented lessons for continuous improvement.

**Primary owner:** Project Manager

**Stakeholders:** _(none listed)_

**Accelerated MVP:** optional when change control is the PO's call

## What this is

The bundled artifact presented to decision-makers for a scope, schedule, or budget change — consolidates the BA's change impact report, Architect's technical impact analysis, cost delta, risk posture, and recommended decision with alternatives.

## Why it matters for the Project Manager

Changes approved on vibes derail baselines. Packaging impact, cost, and recommendation into a single reviewable artifact is how you get decisive answers from Sponsor/PO forums rather than "let me think about it" loops — and how you ensure the baseline updates match the decisions made.

## Definition of Done

- [ ] Change request summary identifies the requestor
- [ ] Change impact report and technical impact analysis are attached
- [ ] Schedule, cost, and scope deltas each state a confidence level
- [ ] Recommendation is stated with at least one alternative
- [ ] Decision record slot is populated with outcome, decider, and date

## What it contains

- Change request summary and requestor
- Change impact report and technical impact analysis attached
- Schedule, cost, and scope deltas with confidence
- Risk delta and security/compliance assessment
- Recommendation and alternatives
- Decision record slot (approved/rejected/deferred, by whom, when)

## Inputs you rely on

- Change impact report and technical impact analysis
- Baselined schedule and budget burn report
- Security control requirements (if the change affects controls)
- RAID log and feature/epic status report
- Stakeholder comms plan for notification post-decision

## Who consumes it

- Product Sponsor and PO — make the call
- Architect and Security — attest to technical/control impact
- Finance — updates budget authorization
- Project Manager — updates baseline and downstream plans

## Common pitfalls

- Package presented without a clear recommendation — forum stalls
- Missing alternatives — stakeholders have no basis to trade off
- Approved but never updated in plan/baseline — drift disguised as "approved change"
- Compliance and security bypassed when time pressure is high
