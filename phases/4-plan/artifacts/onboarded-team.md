# onboarded team

_Produced by: Team onboarding & ramp-up_

**Business outcome supported:** stand up the team, environments, pipeline, test strategy, SLOs, and work-ready backlog so engineering can begin productively on day one.

**Primary owner:** Project Manager

**Stakeholders:** _(none listed)_

**Accelerated MVP:** optional when the team already knows the domain and tools

## What this is

Evidence the full delivery team — engineers, QA, designers, and specialists — is in-seat, has access, knows the domain, and can start shipping on day one of Build. Captured as a checklist, not a narrative.

## Why it matters for the Project Manager

Every day engineers spend waiting for access, context, or kit is a day of baseline slip you can't recover. Treating onboarding as an artifact you drive to completion — not a background activity — is how you protect the schedule and hit the Build-phase ramp curve the baseline assumes.

## Definition of Done

- [ ] Roster lists roles, start dates, and allocation percentage for every member
- [ ] Access checklist (repos, environments, ticketing, docs, comms, cloud) is complete per person
- [ ] Required trainings (security, privacy, domain) are recorded as completed
- [ ] Each member's local dev environment is verified by a first green PR
- [ ] Introductions to upstream and downstream teams are logged
- [ ] Known risks on specific members (partial allocation, notice periods) are called out

## What it contains

- Roster with roles, start dates, and allocation percentage
- Access checklist: repos, environments, ticketing, docs, comms, cloud accounts
- Required training completed (security, privacy, domain briefings)
- Local dev environment verified by a first green PR
- Introductions made to upstream/downstream teams
- Known risks on specific team members (partial allocation, notice periods)

## Inputs you rely on

- Skills requirement input and approved resource commitment
- Provisioned environments (dev/CI readiness)
- Architecture and design package for context briefings
- Working CI/CD pipeline for the "first green PR" milestone
- HR/IT onboarding workflow commitments

## Who consumes it

- Product Sponsor — sees team is materially ready before committing to baseline
- Engineering Manager — confirms each person is productive
- Security & Compliance — verifies required trainings complete
- Release Manager and SRE — know who will be paged and when

## Common pitfalls

- Calling the team "onboarded" when access is partial — week-one silent blockers
- Treating contractors and vendor staff as exceptions and leaving them under-oriented
- Skipping domain briefings — engineers write code against the wrong mental model
- No first-PR gate so onboarding "completion" is self-reported
