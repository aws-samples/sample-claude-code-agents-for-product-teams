# delivery-ready work items

_Produced by: Backlog decomposition (features → work items)_

**Business outcome supported:** stand up the team, environments, pipeline, test strategy, SLOs, and work-ready backlog so engineering can begin productively on day one.

**Primary owner:** Product Owner

**Stakeholders:** Business Analyst

**Accelerated MVP:** required — items must be actionable

## What this is

Features broken down to the grain engineering can actually pull into an iteration — small, independently valuable, with acceptance criteria and dependencies resolved. The output of backlog decomposition, not the raw backlog.

## Why it matters for the Product Owner

Velocity evaporates when engineers hit items they can't finish in-iteration because they're too big, too fuzzy, or blocked. You own the sequence; the BA owns the rigor; together this artifact is where feature-level intent becomes work the team will complete and you can accept.

## Definition of Done

- [ ] Every item has a clear title and problem statement at story/task grain
- [ ] Each item lists testable acceptance criteria
- [ ] Each item carries an engineering size estimate
- [ ] Dependencies (upstream items, external teams, data, design) are called out
- [ ] Each item links back to parent feature/epic and source requirement
- [ ] Definition-of-ready signoff state is recorded

## What it contains

- Work items at story/task grain with clear titles and problem statements
- Testable acceptance criteria per item
- Sizing from engineering (story points, t-shirt, or hours)
- Dependencies called out (upstream items, external teams, data, design)
- Links back to parent feature/epic and requirement for traceability
- Definition-of-ready signoff state

## Inputs you rely on

- Prioritized backlog (what to decompose next)
- Testable acceptance criteria from the BA
- Design specifications and interactive prototype
- Effort estimates from developers
- Enabler backlog for technical prerequisites

## Who consumes it

- Developers — pull these into iterations and implement
- QA/Tester — derive test cases from the acceptance criteria
- Project Manager — uses count and size for iteration planning
- Business Analyst — maintains traceability and clarifies intent during Build

## Common pitfalls

- Items too large to finish in one iteration — partial completion every sprint
- Acceptance criteria that describe UI rather than observable behavior
- Missing non-functional acceptance (perf, a11y, security) so those get deferred
- Decomposing too far ahead — items go stale before they're pulled
