# iteration plan

_Produced by: Iteration / milestone planning_

**Business outcome supported:** stand up the team, environments, pipeline, test strategy, SLOs, and work-ready backlog so engineering can begin productively on day one.

**Primary owner:** Project Manager

**Stakeholders:** _(none listed)_

**Accelerated MVP:** slim — a next-iteration plan plus a rough second is enough

## What this is

The near-term iteration schedule laid over the technical milestone plan — which iterations target which capabilities, capacity per iteration, demo and review cadence, and hand-off dates with other teams.

## Why it matters for the Project Manager

The iteration plan is where the baseline meets reality week to week. You produce it so every role can see what they owe in the next two to four iterations and so slip shows up early enough to react rather than explain.

## Definition of Done

- [ ] Iteration dates, length, and ceremonies are documented
- [ ] Capacity per iteration is listed by role or skill
- [ ] Target scope per iteration names features, enablers, and milestones
- [ ] Demo audience and review cadence are specified
- [ ] External hand-off or integration dates in-scope are captured
- [ ] Known time-off, holidays, and capacity risks are flagged

## What it contains

- Iteration dates, length, and ceremonies
- Capacity per iteration by role/skill
- Target scope per iteration (features, enablers, milestones)
- Demo audience and review cadence
- External hand-off or integration dates in-scope
- Known time-off, holidays, and risk to capacity

## Inputs you rely on

- Baselined schedule and technical milestone plan
- Prioritized backlog and delivery-ready work items
- Resource schedule and onboarded team
- Dependency list
- Effort estimates

## Who consumes it

- Developers and Designers — plan work commitments in each iteration
- Product Owner — confirms sequence matches value priorities
- Project Manager — tracks iteration-on-iteration progress against the baseline
- Stakeholders — know when to expect demos and what's on deck

## Common pitfalls

- Planning past the planning horizon — iterations four+ out are fiction
- Ignoring declared capacity and overloading iterations to hit the baseline on paper
- Not reserving capacity for defects, toil, and support — velocity collapses mid-Build
- Demo audiences never defined so the ceremony drifts into a retro
