## What this is

A clickable prototype — low- to mid-fidelity — that simulates the core flows closely enough to let users and stakeholders experience them before build. It is a research instrument, not a product, and its disposable nature is a feature.

## Why it matters for the UI/UX Designer

You build prototypes to test the riskiest flows with real users while changes cost minutes, not weeks. Evidence from prototype testing is how you retire "nice" arguments with "here is what happened when eight users tried it."

## Definition of Done

- [ ] Core happy-path flows are clickable end-to-end.
- [ ] At least a few key alternate paths and error states are included.
- [ ] Content is realistic, not placeholder / lorem ipsum.
- [ ] Known gaps (flows not yet prototyped) are called out.
- [ ] Version and what is being tested in each iteration are stamped on the prototype.

## What it contains

- Core happy-path flows interactive end-to-end
- A few key alternate paths and error states
- Realistic content, not placeholder text
- Instructions for test facilitators or async testers
- Known gaps (flows not yet prototyped)
- Version and what is being tested in each iteration

- Target users — generate [usability findings](usability-findings.md)
- Product Owner and Sponsor — preview the product before build commitment

- Sales & Marketing — sanity-tests the story the product will tell

## Common pitfalls

- Over-polishing a prototype until it looks production-ready and expectations mis-set
- Testing with people who are not the target persona
- Only prototyping the happy path, so edge cases never get user feedback
- Keeping the prototype alive as "truth" after handoff, creating two sources of design
