# usability findings

_Produced by: Usability testing (pre-build)_

**Business outcome supported:** produce a build-ready design package — UX, architecture, data model, APIs, threat model, and integration map — with engineering signed off to build.

**Primary owner:** UI/UX Designer

**Stakeholders:** _(none listed)_

## What this is

The evidence from usability tests on the interactive prototype — task success, errors, confusions, quotes, time-on-task — synthesized into findings and recommendations. Done pre-build so the design can still change cheaply.

## Why it matters for the UI/UX Designer

You own this because subjective "looks good" reviews are not evidence. Usability findings are what you point to when a stakeholder pushes for a change that hurts real users, and the artifact that justifies the design choices you are defending in review.

## Definition of Done

- [ ] Test plan names tasks, personas recruited, and n per segment.
- [ ] Quantitative metrics (task success, errors, time-on-task, SUS-equivalent) are reported.
- [ ] Qualitative themes include supporting user quotes.
- [ ] Each issue has a severity rating.
- [ ] Recommended design changes are listed with priority.

## What it contains

- Test plan: tasks, personas recruited, n per segment
- Quantitative metrics: task success, error rate, time-on-task, SUS or equivalent
- Qualitative themes with supporting quotes
- Severity rating for each issue found
- Recommended design changes with priority
- Known coverage gaps and follow-up test plan

## Inputs you rely on

- [Interactive prototype](interactive-prototype.md) and test scripts
- [User personas](../../1-discover/artifacts/user-personas.md) for recruiting
- [Journey map / service blueprint](journey-map-service-blueprint.md) for task selection
- Accessibility testing partners for assistive-tech sessions

## Who consumes it

- UI/UX Designer (future you) — applies changes to [design specification](design-specification.md) and [visual design mockups](visual-design-mockups.md)
- Product Owner — uses severity to decide what ships in MVP
- Business Analyst — revises requirements or acceptance criteria where behavior is wrong
- Developer — understands why certain patterns are non-negotiable

## Common pitfalls

- Small n with strong conclusions — be honest about what you saw
- Recruiting convenience sample instead of target personas
- Issues catalogued without severity, so Product Owner can't prioritize
- Findings that never make it into design edits — research theater
