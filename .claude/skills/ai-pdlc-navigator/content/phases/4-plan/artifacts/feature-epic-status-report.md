# feature/epic status report

_Produced by: Track features & epics_

**Business outcome supported:** stand up the team, environments, pipeline, test strategy, SLOs, and work-ready backlog so engineering can begin productively on day one.

**Primary owner:** Project Manager

**Stakeholders:** _(none listed)_

## What this is

A rolled-up status view at the feature/epic level — progress against plan, forecast completion, confidence, and top risks per feature. Not a list of tickets; a narrative of where each major deliverable stands.

## Why it matters for the Project Manager

Sponsors and stakeholders don't care about ticket counts — they care about whether the features they committed to customers will ship on time. This artifact is the language you use to report up and the dataset you use to trigger change-control conversations before the slip crystallizes.

## Definition of Done

- [ ] Each feature/epic lists an owner and target iteration/milestone
- [ ] RAG status is accompanied by reasoning, not just a color
- [ ] A forecast completion date is compared against baseline per feature
- [ ] Top risks and mitigations are named per feature
- [ ] Blocking dependencies list owners and ETAs
- [ ] Scope changes since the last report are captured

## What it contains

- Feature/epic list with owner and target iteration/milestone
- RAG status per feature with reasoning, not just color
- Forecast completion date vs. baseline
- Top risks and mitigations per feature
- Dependencies blocking progress, with owners and ETAs
- Scope changes since last report

## Inputs you rely on

- Iteration plan and assigned workitems & daily status
- Prioritized backlog and delivery-ready work items
- RAID log and risk register
- Baselined schedule
- Technical milestone plan for cross-cutting work

## Who consumes it

- Product Owner — trades scope vs. time when a feature is at risk
- Product Sponsor — uses it as a primary input to executive updates
- Stakeholders — know what to tell their customers
- Release Manager — plans release composition around confident features

## Common pitfalls

- RAG without reasoning — "amber" with no explanation is noise, not signal
- Carrying green forecasts long after evidence says otherwise
- Confusing "work done" with "outcome delivered" — features that are code-complete but not accepted
- Report cadence too slow to drive decisions while they still matter
