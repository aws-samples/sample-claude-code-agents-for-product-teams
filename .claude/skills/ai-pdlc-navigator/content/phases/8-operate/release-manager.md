# Release Manager in Operate

Operate is where the next release's integrity starts earning itself. You're not the lead on Operate-phase artifacts, but you maintain the release cadence, track change failure rates, keep rollback capability warm, and integrate Operate learnings into the runbooks and BOM discipline that make Launch repeatable.

## What you do

- Maintain release cadence for bug-fix, security patch, and minor releases.
- Track change-failure rate and mean-time-to-restore so release integrity stays measurable.
- Keep rollback, canary, and progressive-rollout capability operationally warm.
- Integrate operational learnings into runbooks and the deployment playbook.
- Coordinate with Project Manager on the operating cadence for release reviews.

## Artifacts you own

You don't own any artifacts this phase — you contribute to others'.

## Artifacts you contribute to

You don't contribute to others' artifacts this phase.

## Outcomes you drive

You don't drive outcomes this phase — you input into others'.

## Who you work with

- **Site Reliability Engineer** — bi-directional on release-related reliability and rollback.
- **Developer** — bi-directional on release candidate quality and deploy cadence.
- **QA/Tester** — bi-directional on release verification.
- **Security & Compliance** — bi-directional on security-patch release coordination.
- **Project Manager** — bi-directional on operating cadence for releases.

## Handoff into [Iterate](../9-iterate/README.md)

You exit Operate with a tuned release cadence and real change-failure evidence. Carry that into Iterate's retrospective and lessons-learned work — the Operate-phase release data is the highest-signal input you have for improving release integrity on the next cycle.
