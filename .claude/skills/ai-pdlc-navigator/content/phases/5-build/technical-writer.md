# Technical Writer in Build

Build is where your drafts start landing. As features merge, you turn them into user-facing documentation — in-product copy, API reference from auto-generated scaffolds, first-pass user guides. Your job is to keep pace with development so Verify reviews a real draft, not a blank page.

## What you do

- Draft user-facing documentation against merged features so content matures with the product.
- Review auto-generated API docs and fill the gaps where prose matters.
- Refine in-product copy with UI/UX Designer as the UI takes shape.
- Flag documentation debt when undocumented features land so it doesn't pile up for Launch.

## Artifacts you own

You don't own Build-phase artifacts formally — your drafts feed Verify's documentation review and Launch's feature docs / user guides.

## Artifacts you contribute to

- [generated API docs](artifacts/generated-api-docs.md) — Developer owns the generation; you own the prose gaps.

## Outcomes you drive

You don't drive outcomes this phase — you input into others'.

## Who you work with

- **Developer** — generates API scaffolds; you refine and extend.
- **UI/UX Designer** — aligns on in-product copy, tone, terminology.
- **Product Owner** — you flag docs debt affecting Launch readiness.
- **Customer Support** — aligns on what content they need to deflect tickets.

## Handoff into [Verify](../6-verify/README.md)

In Verify you co-own the documentation review with QA/Tester — walking the product with the docs to catch drift. You know Build is done when every accepted increment has a docs draft, API docs compile clean, and nothing critical is silently undocumented.
