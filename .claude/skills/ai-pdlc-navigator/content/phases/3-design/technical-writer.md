# Technical Writer in Design

Design is where you can start doing real work even though no docs ship yet. The IA, journeys, API spec, and data model emerging this phase are the skeleton of your eventual docs architecture. Pair with the UI/UX Designer on language and structure; align your planned docs IA to theirs so users don't hit two different mental models.

## What you do

- Read the [IA spec](artifacts/ia-spec.md), [design specification](artifacts/design-specification.md), [API specification](artifacts/api-specification.md), and [journey map / service blueprint](artifacts/journey-map-service-blueprint.md) — they define your docs IA
- Start drafting a docs outline against the approved scope — surface gaps early
- Capture terminology consistently with the designer's UI copy

## Artifacts you own

You don't own any artifacts this phase — you contribute to others' (see below).

## Outcomes you drive

You don't drive outcomes this phase — you input into others'.

## Who you work with

- **UI/UX Designer** — they're defining IA and UI copy; align your docs IA and voice with theirs
- **Architect** — their API spec and data model feed your reference docs
- **Developer** — you'll need their inline docs and API comments to become generated reference later

## Handoff into Plan

Nothing you produce is a hard handoff into [Plan](../4-plan/README.md), but the docs IA you've sketched should be ready to start producing content when features land in Build. The quality bar is that when Build begins you're already drafting with a clear IA and terminology. You're done when you have a docs IA you'd commit to publicly and a sense of what has to ship at GA.
